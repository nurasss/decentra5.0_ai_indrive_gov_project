# Анализ проекта `adilet_scraper`

## 1. Краткий вывод

Сейчас проект выглядит как хороший рабочий набросок под задачу массового сбора нормативных документов с Adilet, но не как законченный быстрый production-парсер.

Главный плюс: у проекта уже есть реальный результат в данных, и сама задача выбрана удачно.

Главный минус: текущий код в репозитории не совпадает по зрелости с масштабом выгрузки. По состоянию файлов проект трудно воспроизвести, ускорить и безопасно поддерживать.

## 2. Что есть в проекте

- `adilet_scraper/spiders/adilet_spider.py` — основной паук.
- `adilet_scraper/settings.py` — ускоренные настройки Scrapy.
- `adilet_scraper/items.py` — структура item.
- `adilet_scraper/pipelines.py` — очистка текста перед сохранением.
- `output.csv` — маленькая выгрузка.
- `clean_laws.csv` — большая итоговая выгрузка.

Отдельной презентации в папке проекта не найдено: нет `ppt`, `pptx`, `pdf`, `md` с описанием проекта. Поэтому оценка презентации ниже касается только того, как проект представлен в текущем репозитории.

## 3. Сильные стороны проекта

### По задаче

- Задача практичная и ценная: сбор правовых актов — это полезный и понятный датасет.
- У проекта уже есть итоговый артефакт большого объема: `clean_laws.csv` весит около 783 MB.
- По данным видно, что была проделана реальная работа по извлечению текста, а не только тестовый парсинг.

### По архитектурной идее

- Выбран Scrapy, а это правильный стек для быстрого массового парсинга.
- В `settings.py` уже есть попытка оптимизировать скорость через `CONCURRENT_REQUESTS = 32` и отключение лишнего (`COOKIES_ENABLED = False`, `TELNETCONSOLE_ENABLED = False`).
- В проекте предусмотрена логика дозагрузки/продолжения обхода через проверку уже обработанных URL в `output.csv`: [adilet_spider.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/spiders/adilet_spider.py#L24).

### По данным

- `clean_laws.csv` содержит `23153` уникальных URL.
- Полностью пустой текст только у `659` записей, что для такого объема не критично, но требует доработки.
- Коротких текстов меньше 200 символов — `717`, то есть основная масса документов все же извлечена содержательно.

## 4. Слабые стороны проекта

### Критичные

1. Код в репозитории не воспроизводит уже полученный результат.
   В текущем `parse()` виден только `title = response.css('h1::text').get()` без `yield item`, без извлечения основного текста и без экспорта результата: [adilet_spider.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/spiders/adilet_spider.py#L39).

2. Захардкожен абсолютный путь к входному CSV.
   Это ломает переносимость проекта на любой другой компьютер: [adilet_spider.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/spiders/adilet_spider.py#L15).

3. Нет среды запуска и зависимостей.
   В проекте нет `requirements.txt`, `pyproject.toml`, `Pipfile`, `README.md`. Команда `scrapy version` в текущей среде не выполняется, то есть проект нельзя просто взять и запустить.

4. Несогласованность модели данных.
   В `items.py` есть поле `content`: [items.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/items.py#L3).
   В CSV используется колонка `text`.
   В `pipelines.py` очищается `content`: [pipelines.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/pipelines.py#L4).
   Это признак расхождения между старым и текущим кодом.

### Средней тяжести

1. Возобновление парсинга реализовано через полное чтение `output.csv` в память.
   Это решение будет плохо масштабироваться на больших выгрузках: [adilet_spider.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/spiders/adilet_spider.py#L26).

2. Используется `pandas` в точке старта паука для простого чтения URL.
   Для списка ссылок это тяжелее, чем нужно. Для скоростного парсера лучше потоковое чтение CSV стандартным модулем `csv`.

3. На каждый документ идет `print()` в консоль.
   При высокой конкуррентности это создает лишний I/O bottleneck и замедляет процесс: [adilet_spider.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/spiders/adilet_spider.py#L44).

4. Включен очень агрессивный режим запросов без адаптивного контроля.
   `CONCURRENT_REQUESTS = 32`, `DOWNLOAD_DELAY = 0`, `AUTOTHROTTLE` выключен: [settings.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/settings.py#L25), [settings.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/settings.py#L30), [settings.py](/Users/bulat/Desktop/adilet_scraper/adilet_scraper/settings.py#L82).
   Это может быть быстро, но нестабильно и хрупко при изменении поведения сайта.

5. Комментарии частично повреждены по кодировке.
   В коде заметен mojibake, значит были проблемы с сохранением файлов в корректной кодировке.

### Слабые стороны сопровождения

- Нет тестов на селекторы.
- Нет проверки, что страница действительно содержит текст закона, а не ошибку/редирект/служебную страницу.
- Нет явного логирования причин пустых документов.
- Нет метрик качества: сколько 200/404/timeout/empty text.
- Нет README с командой запуска и описанием пайплайна.
- Нет разбиения на этапы: сбор ссылок, скачивание, очистка, валидация.
- Проект не открыт как git-репозиторий, поэтому сложно отслеживать историю изменений.

## 5. Что мешает сделать парсер действительно быстрым

Сейчас узкое место не только сеть, но и организация процесса.

### Основные тормоза

1. Полное чтение уже собранного CSV для дедупликации.
   На больших файлах это лишняя память и лишнее время на старт.

2. Лишний `print()` на каждую страницу.
   При тысячах документов это заметно бьет по скорости.

3. `pandas` на старте вместо легкого потокового чтения.

4. Нет оптимизированного формата хранения промежуточного состояния.
   CSV удобен для выгрузки, но неудобен как рабочее хранилище состояния.

5. Нет разделения на быстрый crawler и постобработку.
   Очистку текста и нормализацию лучше частично выносить после скачивания.

## 6. Что мешает сделать парсер отличным

Отличный парсер — это не просто быстрый парсер. Он должен быть:

- воспроизводимым;
- устойчивым к сбоям;
- понятным для поддержки;
- проверяемым по качеству данных;
- легко перезапускаемым;
- масштабируемым.

Сейчас у проекта есть скорость как идея, но пока не хватает:

- воспроизводимости;
- надежной схемы данных;
- контроля качества;
- удобного запуска;
- технической презентации проекта.

## 7. Оценка качества текущих данных

### Что хорошо

- `23153` уникальных документов в `clean_laws.csv` — это уже серьезный результат.
- У большинства записей присутствуют `title` и `text`.
- Формат таблицы простой и удобный для последующей обработки.

### Что настораживает

- `659` документов без текста.
- `717` документов очень короткие.
- Непонятно, являются ли эти случаи валидными короткими указами или это ошибки селектора/загрузки.
- Нет отдельного поля со статусом извлечения: `ok`, `empty`, `not_found`, `redirect`, `parse_error`.

## 9. Приоритетный план улучшения

### Этап 1. Сделать проект воспроизводимым

- Добавить `README.md`.
- Добавить `requirements.txt` или `pyproject.toml`.
- Убрать абсолютные пути.
- Передавать входной CSV через аргумент паука или settings.

### Этап 2. Сделать код консистентным

- Привести к одной схеме поля `url`, `title`, `text`.
- Обновить `items.py`, `pipelines.py`, экспорт и паука под одну модель.
- Восстановить реальную логику `parse()` в коде репозитория.

### Этап 3. Сделать парсер быстрым

- Убрать `print()` на каждый ответ.
- Заменить `pandas` на потоковое чтение входного CSV.
- Хранить уже обработанные URL не в полном `output.csv`, а в отдельном компактном state-файле или SQLite.
- Включить разумный баланс между скоростью и стабильностью: например, не только `CONCURRENT_REQUESTS`, но и AutoThrottle/ограничения на ошибки.

### Этап 4. Сделать парсер надежным

- Сохранять статус обработки по каждому URL.
- Логировать пустые документы отдельно.
- Добавить retry-логику с диагностикой по кодам ответа.
- Проверять, что селектор нашел основной контейнер документа.

### Этап 5. Сделать проект сильным для защиты/презентации

- Добавить бенчмарки.
- Показать качество данных в цифрах.
- Описать архитектуру и компромиссы.
- Подготовить короткий demo-flow запуска.
